# AGENTS.md ベストプラクティス

## 目的

`AGENTS.md` は、エージェントがそのリポジトリで最初に知るべき実務情報だけを短く渡すための入口と考える。
読み始めてすぐに「何を実行し、どこを見て、何を避けるべきか」が分かる状態を目指す。

## ルートに優先して書く内容

- リポジトリの一行説明
- 日常的に使うコマンド
- 最低限の検証手順
- 重要ディレクトリや編集対象の見つけ方
- 破壊的操作、承認、生成物などの安全上の注意
- 追加資料への短い案内

最初の数十行で着手に必要な情報へ到達できる構成を優先する。

## 後ろへ送る内容

- 一部のタスクでしか使わない詳細手順
- サブシステム固有の深い説明
- 長い設計背景
- 例外ケースの網羅
- 既存の `README.md` や `docs/` に既にある詳細

これらは `docs/agents/*.md` に分割し、ルート側には一行説明付きリンクだけ残す。

## 書くべきこと

- 実在するコマンド、実在するパス、実在するワークフロー
- エージェントが毎回迷いやすい判断
- リポジトリ固有の制約
- よくある失敗を避けるための短い注意
- 既存ドキュメントを読む順番

## 書くべきでないこと

- 抽象的な一般論だけの心得
- 人間向け README の全文再掲
- 未確認のコマンドや推測
- すべてのケースを網羅しようとする長文
- 特定のモデルやツールでしか意味を持たない記法への依存

## 長さの目安

- ルート `AGENTS.md` は理想 60-150 行
- 300 行を超えそうなら分割を検討
- 補助文書は必要なときだけ読む前提で分離する

長さそのものより、先頭から読んだときの着手速度を重視する。

## 分割のルール

- ルートは入口、補助文書は詳細という役割を崩さない
- 分割先は `docs/agents/*.md` に寄せる
- 分割先のリンクには「何が書いてあるか」を一言添える
- 分割しても、ルート単体で基本作業を始められる状態を保つ

## 生成時のチェックリスト

- 主要コマンドは実在するか
- パスは実在するか
- 既存 docs と重複していないか
- ルートの前半に最重要情報が集まっているか
- 長い詳細をルートに残しすぎていないか

## 生成後レビューの観点

- 先頭 30-50 行で着手に必要な情報へ到達できるか
- すべてのコマンドとパスが実在するか
- ルートにある各項目が本当に高頻度で普遍的か
- 既存 `README.md` や `docs/` の焼き直しが残っていないか
- 長い詳細や低頻度手順をルートに残しすぎていないか
- 壊れやすいファイル配置や古くなりやすい断定を書きすぎていないか
- 特定ツールや特定モデルでしか意味を持たない癖を混ぜていないか
- 読み手が「次に何を実行するか」を迷わない順序になっているか

yes/no で一度通し、no が出た項目は原則その場で修正する。

## 自己修正の優先順位

- 事実誤認
- ルートの過積載
- 既存 docs との重複
- 順序の悪さ
- 文言の抽象度が高すぎる箇所

迷ったら「足す」より「削る」「外へ逃がす」を優先する。
