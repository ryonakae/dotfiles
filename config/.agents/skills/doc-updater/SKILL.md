# doc-updater

## 概要

セッションの変更を見て、ドキュメントへの反映が必要か判断し、必要な更新だけを行う。staged change があればそれを対象にし、無い場合は `git diff HEAD` と未追跡ファイルにフォールバックする。実行環境が skill の明示呼び出しをサポートしていればそれを使い、サポートしていなければこのファイルを読んで同じ手順を inline で実行する。

## 前提

- 通常は staged change だけを対象にする
- `git diff --cached` が空の場合（stop hook など、まだステージングされていない文脈）は `git diff HEAD`（staged + unstaged の全変更）と未追跡ファイルにフォールバックする
- シンボリックリンクは直接編集しない
- 差分は最小限に留める
- コミットや push は行わない
- ドキュメントを更新したら `git add` でステージングに追加する
- 呼び出し元から staged file list と rename/delete context が渡されている場合は、それを優先して使う
- 呼び出し元からセッション中の失敗や修正で残すべき学びが渡された場合は反映する
- 独立したコンテキストで実行された場合は、最後に呼び出し元へ更新ファイル、変更概要、更新不要と判断した理由を簡潔に返す

## スキップしてよい条件

- 呼び出し元が「このセッションで既にドキュメント更新済み」と明示している
- 呼び出し元が「更新不要と判断済み」と明示している
- ドキュメントファイルがすでに staged されていて、今回のコード変更で追加更新が不要と判断できる
- 対象ファイル（staged またはフォールバック時の変更ファイル）がすべて `docs/` 配下または `*.md` で、コード変更が無い場合

スキップする場合は「ドキュメント更新は不要です」とだけ返して終了する。

## 1. ドキュメント構造の検出

ルート直下のエージェント指示ファイル（`AGENTS.md`, `CLAUDE.md`, `GEMINI.md`）のうち、シンボリックリンクでない実ファイルを特定する。

- `README.md` は人間向けドキュメントとして別に確認する
- symlink の場合はリンク先の実ファイルだけを編集対象にする

確認コマンドの例（実行環境に応じて適した方法を使う）:

```bash
find . -maxdepth 1 \( -name "AGENTS.md" -o -name "CLAUDE.md" -o -name "GEMINI.md" \) ! -type l
```

分割ドキュメント構造がある場合は `docs/agents/` と `docs/agent/` 配下の `.md` ファイルも確認する。

- 両方ある場合は、ルートからリンクされている側を優先する
- トピックは「ファイル名」「H1 見出し」「ルートからのリンク文言」の 3 点で把握する

## 2. 変更の把握

### 通常（staged change あり）

`git diff --cached` に出力がある場合は staged change だけを対象にする。

```bash
git diff --cached --name-only --diff-filter=ACMRD
git diff --cached
git diff --cached --diff-filter=R --name-status
git diff --cached --diff-filter=D --name-only
```

### フォールバック（staged change なし）

`git diff --cached` が空の場合は以下の手順で収集する。

1. 追跡済みファイルの変更（staged + unstaged）を取得する:

```bash
git diff HEAD --name-only --diff-filter=ACMRD
git diff HEAD
git diff HEAD --diff-filter=R --name-status
git diff HEAD --diff-filter=D --name-only
```

2. 未追跡ファイル（`git status --short` で `??` のもの）を列挙し、ファイル内容を直接読んで「全体が新規追加」として扱う:

```bash
git status --short | grep '^??' | awk '{print $2}'
```

以下の情報を収集する（通常・フォールバック共通）:

- 変更ファイル一覧（追加・変更・削除・リネームを含む）
- 各ファイルの差分内容（未追跡ファイルはファイル本文）
- リネームされたファイルの旧パスと新パス
- 削除されたファイルのパス（ドキュメント内に参照が残っていないか確認するため）

## 3. 更新要否の判定

以下は更新する。

- 機能追加、API 変更、振る舞い変更
- エージェント、スキル、フック、CLI workflow の変更
- 依存関係ファイルの変更
- ビルド、テスト、実行コマンドの変更
- 設定ファイルの変更
- ファイルやディレクトリのリネーム、削除
- README や AGENTS 系に書かれている手順、構成、制約が古くなる変更

以下は更新不要。

- 変数名変更、関数抽出、内部リファクタリングだけで仕様が変わらない
- テストやコメントだけの変更
- ドキュメントファイルだけの変更

迷ったら更新する側に倒す。

## 4. 更新先の決定

分割構造がない場合はルートファイルに書く。

分割構造がある場合は次で振り分ける。

- トピック固有の詳細は該当する分割ファイル
- プロジェクト全体に関わる変更はルートファイル
- 分割ファイルの追加、削除、リネームがある場合はルートのポインタリンクも更新
- 新トピックは既存の近い分割ファイルへの追記を優先し、難しければ新規ファイルを作ってルートにポインタを追加

## 5. 更新時の原則

- 既存の構造とトーンを維持する
- コードを長く複製せず、ファイルパスやコマンドで示す
- 存在しないパス参照は更新または削除する
- エージェント指示ファイルは WHY / WHAT / HOW を優先する
- 自明な説明や重複した指示は増やさない
- symlink を壊さず、実ファイルだけを編集する

## 6. 検証

更新後は次を確認する。

- ドキュメント内のファイルパス参照が実在する
- エージェント指示ファイルを更新した場合は 300 行未満に収まっている
- symlink を直接編集していない

## 7. 出力

最後に次を簡潔に報告する。

- 編集対象として選択したファイル
- 更新したファイル名と変更概要
- 更新をスキップしたファイルとその理由
- エージェント指示ファイルを更新した場合は現在の行数
