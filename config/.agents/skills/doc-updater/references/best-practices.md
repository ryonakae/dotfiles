# ドキュメント更新のベストプラクティス

## 目的

ドキュメントを「living documentation」として維持する。追記するだけでなく、既存記述の更新・統合・削除も同時に行い、膨張を防ぐ。

## 追記時のルール

- 追記する前に、同セクション内に類似・重複する既存記述がないか確認する
- 重複がある場合は追記ではなく既存記述を更新・統合する
- 削除・リネームされた機能やファイルに関する既存記述を探して除去する
- 具体的な実装詳細（関数名、引数など）より、振る舞いやコマンドレベルで書く

## ポインタ > コピー

コードスニペットをドキュメントに埋め込まない。代わりにファイルパスや `file:line` で参照する。
埋め込んだコードはすぐ陳腐化し、ドキュメントの信頼性を下げる。

## ツールで強制できることは書かない

linter、formatter、型チェッカーで自動的に担保されるルールはドキュメントに転記しない。
ツール設定が正であり、ドキュメントに書くと二重管理になる。

## 書くべきでないこと

- コードから自明な情報
- 既存 `README.md` や `docs/` の焼き直し
- 抽象的な一般論だけの心得
- すぐ陳腐化する実装詳細（内部関数名、引数の型など）
- リポジトリから確認できない推測

## 行数・構造の基準

- ルートのエージェント指示ファイル（`AGENTS.md`, `CLAUDE.md` など）は 300 行未満を維持する
- 更新後に閾値を超えた場合は出力で警告する
- 長い詳細は分割ファイルへの切り出しを提案する（実行はしない）
- 分割よりも、まず既存記述の圧縮・統合で行数を減らせないか検討する

## レビュー観点

更新した箇所とその周辺について、以下を確認する。

- 追加した記述が周辺の既存記述と重複していないか
- パスやコマンドが実在するか
- 既存の文体・トーンを壊していないか
- コードスニペットを埋め込んでいないか（ポインタで代替できないか）
- ツールで強制できるルールを書いていないか

問題があればその場で修正する。

## 修正の優先順位

1. 事実誤認（存在しないパス・コマンドの記述）
2. 既存記述との重複
3. 陳腐化しやすい実装詳細の混入
4. 文体・トーンの不一致
5. 冗長な表現

迷ったら「足す」より「削る」「統合する」を優先する。
