# commit-push 用 doc-updater

## 概要

`commit-push` スキルのステップ2で使う、コミット前のドキュメント更新手順。対象は `git diff --cached` に含まれる変更だけで、Stopフック向けのセッション学習やサブエージェント前提の処理は含めない。

## ステップ1: ドキュメント構造の検出

### 1a. ルート直下のエージェント指示ファイル

1. プロジェクトルートで次を実行し、実ファイルのエージェント指示ファイルを確認する

```bash
find . -maxdepth 1 \( -name "AGENTS.md" -o -name "CLAUDE.md" -o -name "GEMINI.md" \) ! -type l
```

2. 出力された実ファイルだけを編集対象候補にする
3. シンボリックリンクは直接編集しない
4. `README.md` は人間向けドキュメントとして別に確認する
   - `README.md` が symlink ならリンク先の実ファイルだけを編集する

### 1b. 分割ドキュメント構造の検出

```bash
find ./docs/agents ./docs/agent -name "*.md" -type f 2>/dev/null || true
```

5. 分割ファイルが存在する場合、「ファイル名 + H1見出し + ルートからのリンク文言」の3点でトピックを把握する
6. 両方（`docs/agents/` と `docs/agent/`）が存在する場合はルートからリンクされている側を優先する
7. `docs/` 配下の一般ドキュメント（`docs/setup.md`、`docs/architecture.md` 等）は今回の staged 変更に直接関係するファイルだけを確認する

## ステップ2: 変更内容の把握

1. 次のコマンドで staged 変更を確認する

```bash
git diff --cached --name-only --diff-filter=ACMRD
git diff --cached
git diff --cached --diff-filter=R --name-status
git diff --cached --diff-filter=D --name-only
```

2. リネームされたファイルは旧パスと新パスの両方を把握する
3. 削除されたファイルは、ドキュメント内に残っている参照がないか確認する

## ステップ3: 更新要否の判定

以下に当てはまる場合は更新する:

- 機能追加、API変更、振る舞い変更
- エージェント、スキル、フック、CLI ワークフローの変更
- 依存関係ファイルの変更
- ビルド、テスト、実行コマンドの変更
- 設定ファイルの変更
- ファイルやディレクトリのリネーム、削除
- README や AGENTS 系に書かれている手順、構成、制約が古くなる変更

以下だけなら更新不要:

- 変数名変更、関数抽出、内部リファクタリングだけで仕様が変わらない
- テストやコメントだけの変更
- ドキュメントファイルだけの変更

迷ったら更新する側に倒す。更新不要なら、その判断を短く記録して次へ進む。

## ステップ3.5: 更新先の決定

更新先は排他ではなく、1つの変更が複数ファイルに影響することがある。

**分割構造がない場合**: 従来通りルートファイルに書く。

**分割構造がある場合（ステップ1bで検出）**:

| 更新内容 | 更新先 |
|----------|--------|
| トピック固有の詳細（設計判断、注意点、手順等） | 該当する分割ファイル |
| プロジェクト全体に関わる変更（概要、テックスタック、主要コマンド） | ルートファイル |
| 分割ファイルの追加・削除・リネーム | ルートファイルのポインタリンクも同時更新 |
| 新トピック | まず既存の近い分割ファイルへの追記を検討 → 不可なら新規ファイル作成 + ルートにポインタ追加 |

トピック照合は「ファイル名」「H1見出し」「ルートからのリンク文言」の3点で行う。新規ファイルは既存の命名パターンに合わせ1トピック1ファイルとする。

更新実行前に「更新先ファイル一覧」と「各ファイルの更新理由」を明示してから編集に進む。

## 記述規約

エージェント指示ファイル（AGENTS.md / CLAUDE.md 等）を更新する際はこの規約に従う。

### 基本原則

1. **WHY / WHAT / HOW を定義する**
   - WHAT: テックスタック、プロジェクト構造、コードベースの全体像
   - WHY: プロジェクトの目的、各コンポーネントの役割、設計の理由
   - HOW: テスト・型チェック・ビルドなど変更の正しさを検証する手段。検証コマンドは最重要 — これがないとLLMは自分の変更を検証できない
2. **事実形式で書く**: 「〜してください」→「〜を使う」。禁止事項は「トリガー＋アクション」形式
   - 箇条書き・テーブル・見出しで構造化し、長い文章のブロックを避ける
   - 重要なルールには「IMPORTANT」等の強調を使える。ただし多用は逆効果
3. **段階的開示**: 普遍的な情報だけ直接書き、詳細は別ファイルにポインタで示す
4. **コードスタイルはツールに任せる**: リンター/フォーマッターで検出できるルールは書かない。デフォルトから外れるスタイルのみ短く記載する
5. **各行を吟味する**: 「これを削除するとLLMが間違うか？」— Noなら削除。膨れた指示ファイルはLLMが指示を無視する原因になる

### 含めるべきもの / 除外すべきもの

| ✅ 含める | ❌ 除外する |
|-----------|------------|
| LLMが推測できないコマンド | コードを読めば分かる事実 |
| デフォルトと異なるスタイルルール | 標準言語規約・コードスタイル詳細 |
| テスト指示と推奨テストランナー | 詳細APIドキュメント（→リンク） |
| リポジトリのエチケット（ブランチ命名、PR規約）| 頻繁に変わる情報・バージョン番号 |
| プロジェクト固有のアーキテクチャ決定 | 長い説明やチュートリアル |
| 開発環境の癖（必須環境変数） | ファイルごとのコードベース説明 |
| 一般的な落とし穴や明白でない動作 | 自明なプラクティス（「きれいなコードを書く」等）|
| 設計判断の理由・トレードオフ | 特定タスク限定の手順（→別ファイルに分離）|

### 分量と構造

- 60〜300行（短いほど良い）。フロンティアモデルでも150〜200個の指示が限界
- 閾値: 200行超→分離を検討、250行超→分離を強く推奨し警告を出力、300行超→分離を実施
- 最も重要な情報を冒頭、補足は末尾（Lost in the Middle対策）
- 推奨セクション順:
  1. プロジェクト概要（一行説明）
  2. テックスタック
  3. プロジェクト構成（必要な場合のみ）
  4. 開発・検証コマンド（ビルド/テスト/lint/dev）
  5. 設計方針（なぜその構成か）
  6. 注意点（プロジェクト固有の罠。トリガー＋アクション形式）
  7. 詳細ドキュメントへのポインタ（存在する場合のみ）
- コードスニペットは含めない → `file:line` 参照を使う

### プルーニング

情報追加時は既存内容を見直し、以下を削除する:
- コードから自明な事実（LLMはコードを直接読める）
- リンターで検出可能なスタイルルール
- 解決済みの注意点・古い情報
- 同じ意図の重複した指示

## ステップ4: 更新方針

- 既存の構造とトーンを維持する
- 差分は最小限に留める
- コード内容を長く複製せず、ファイルパスやコマンドで示す
- 存在しないパス参照は更新または削除する
- AGENTS 系を更新する場合は、プロジェクト概要、主要コマンド、重要な注意点を優先する
- README を更新する場合は、概要、セットアップ、使い方、構成、制約のうち変更に関係する箇所だけを直す
- 分割構造がある場合、ルートファイルにはポインタリンクと全体共通情報のみを書き、トピック詳細は分割ファイルに書く
- リネーム・削除が発生した場合は、分割ファイル内の相互参照とルートからのポインタリンクも更新対象に含める

## ステップ5: 検証と再ステージ

1. ドキュメント内のファイルパス参照が実在するか確認する（`ls` 等で確認）
2. エージェント指示ファイルを更新した場合、300行未満か確認する（`wc -l` で確認）
3. 対象ファイルを `git add` で再ステージする
4. 変更結果が staged diff に反映されていることを `git diff --cached --name-only` で確認して `commit-push` のステップ3に戻る

## 出力

更新内容のサマリーを以下の形式で返す:
- 編集対象として選択したファイル
- 更新したファイル名と変更概要
- 更新をスキップしたファイルとその理由（あれば）
- エージェント指示ファイルを更新した場合は現在の行数
