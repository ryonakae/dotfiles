# Core Principles

## 使いどころ

- 画面の方向性がまだ曖昧なとき
- ブランド表現と OS 慣習がぶつかるとき
- Flutter / React Native でどこまでプラットフォーム差分を残すか迷うとき

## 判断の優先順位

1. 可用性・アクセシビリティ
2. OS 慣習と学習コスト
3. ブランド世界観

ブランドは強く出してよいが、可読性、主要操作、移動のわかりやすさ、誤操作防止を壊さない。

## 最初に整理すること

- 誰が使うか
- 何を最短で達成したいか
- どの頻度で使うか
- 失敗したときのコストは大きいか
- 移動中、片手操作、短時間操作が前提か

## iOS と Android の共通原則

- 主目的を 1 画面 1 つに寄せる。
- 操作対象は視覚的に分離し、誤タップを避ける。
- 情報は段階的に見せる。最初から全部を詰め込まない。
- 状態変化は明確に示す。読込中、完了、失敗、無効状態を曖昧にしない。
- ナビゲーションは現在地と戻り方を迷わせない。

## iOS で強めに意識すること

- 余白と階層で静かに整理する。
- システム標準の移動感やジェスチャ期待を壊さない。
- 情報量より読みやすさと文脈の自然さを優先する。
- 過度な装飾より、明快さと一貫性を重視する。

## Android で強めに意識すること

- 階層、状態、アクションの関係を明示する。
- リスト、検索、フィルタ、操作密度が高い体験でも破綻しない構造にする。
- 状態の変化や行動結果を明確に返す。
- 大画面や可変幅でも破綻しないレイアウトを意識する。

## Flutter / React Native での考え方

- まず「どの OS の体験期待に寄せるか」を決める。
- 両 OS 共通 UI にするなら、共通化の理由を持たせる。単なる開発都合で均一化しない。
- 重要画面だけ iOS / Android の差分を残し、周辺画面は共通化する設計も許容する。

## ブランドを入れる場所

- 色
- タイポグラフィの緊張感
- アイコンやイラストのトーン
- モーションの速さと性格
- 文言の語彙と態度

主要ナビゲーション、フォームの使い方、戻る/閉じる/保存などの基本操作は、ブランドのために崩しすぎない。
