# Pattern Selection

## ナビゲーション

- 主要カテゴリが 3〜5 個で横断移動が多いなら、タブ系の構造を第一候補にする。
- 階層移動が深く、一覧から詳細へ進む流れが中心なら、スタック型を基本にする。
- フィルタや補助操作を一時的に見せたいなら、シートやドロワーで分離する。
- グローバルな作成操作が主役なら、主要導線として強調する。ただし一覧閲覧と競合させない。

## 一覧と詳細

- 一覧では「比較しやすさ」を優先する。装飾よりスキャンしやすさを取る。
- 詳細では「次に何ができるか」を明確にする。読むだけなのか、編集や購入まで続くのかを分ける。
- カードは万能ではない。区切りが必要なときだけ使い、ただの囲み込みで密度を下げない。

## フォーム

- 入力順は利用者の思考順に合わせる。
- 入力中に直せるエラーは遅らせず示す。ただし赤字だらけで圧迫しない。
- 必須 / 任意、フォーマット、補助説明、失敗理由を曖昧にしない。
- キーボード表示時の視認性と送信導線を必ず考える。

## 検索・絞り込み

- 検索が主要タスクなら、隠さず早い位置に置く。
- フィルタは「結果がどう変わるか」を理解できる粒度にする。
- 高頻度の絞り込みは常設し、低頻度の詳細条件は段階的に出す。

## オンボーディング

- 初回説明は短くする。価値、次の行動、権限理由だけに絞る。
- 権限要求は機能価値と結びつけて出す。
- ブランド紹介だけで数画面を消費しない。

## 空状態・エラー状態・読込中

- 空状態では「なぜ空か」と「次に何をすればよいか」を必ず示す。
- エラー状態では、利用者の責任に見えない書き方を使う。
- 読込中は時間感覚を鈍らせない。長い処理には進行感や代替行動を与える。
- 再試行、戻る、問い合わせなど、次の安全な行動を置く。

## 確認と破壊的操作

- 取り消せる操作なら、確認ダイアログより undo を優先して検討する。
- 取り返しがつかない操作だけ、確認を強くする。
- ボタン文言は「OK」より、何が起きるかを直接書く。

## モーダル / シート / 画面遷移

- その場で完了する短いタスクはモーダルやシートを使う。
- 長い入力や複雑な分岐があるなら、独立画面に出す。
- 閉じる方法と保存タイミングを曖昧にしない。

## よくある誤り

- ナビゲーションとアクションを同じ場所で競合させる
- すべてをカード化して比較しにくくする
- 状態ごとの見え方を設計せず、正常系の画面だけ作る
- フォーム補助文をプレースホルダーだけに押し込む
