# DESIGN.md ベストプラクティス

## 目的

DESIGN.md は、エージェントが UI を実装するときに参照する唯一のデザイン参照ファイル。
入力ソースから抽出したデザイントークンを、エージェントが迷わず使えるトークンレベルの仕様書に変換するのが目的。

「ルール集ではなく参照順を固定するファイル」であることを意識する。
詳細すぎる記述は更新コストを上げ、古くなると害になる。

## 書くべきこと

- 具体値（hex, px, font-family, font-weight）を伴うトークン
- セマンティック名と機能的役割（何に使うか）
- 技術値の自然言語変換（"rounded-full" → "ピル型(9999px)"）
- ブランドの視覚的印象（Visual Theme & Atmosphere）
- コンポーネントのインタラクション状態（hover/focus/disabled）

## 書くべきでないこと

- 修飾なしの一般用語（"blue" だけ、"rounded" だけ）
- デザインシステム全文の転記
- 実装コード（CSS クラス名、Tailwind クラスリスト等）そのもの
- バイナリアセット（ロゴ、アイコン）のインライン記述（ファイルパスへの参照に留める）
- 入力ソースから確認できない推測
- 古くなりやすい一時的な値（キャンペーン色など）

## 長さの目安

- 理想: 150-300行
- 500行を超えそうな場合は、詳細を別ファイルに委譲してDESIGN.mdには参照先リンクを残す
- セクションごとの長さは均等でなくてよい（Color Palette は長くなりやすい、Depth は短くてよい）
- 先頭を読むだけで主要な色・フォント・スペーシングが把握できる構成を優先する

## 生成時のチェックリスト

- [ ] 9セクション全てが存在するか
- [ ] 全ての色に hex 値・セマンティック名・機能的役割があるか
- [ ] フォント名は実在するか（推測で書いていないか）
- [ ] 技術値が自然言語で補足されているか（角丸・影・フォントウェイト）
- [ ] 曖昧な記述（"blue"、"rounded" だけ）が残っていないか
- [ ] 入力ソースに基づかない推測が混入していないか
- [ ] 既存 DESIGN.md がある場合、差分更新になっているか

## 生成後レビューのルブリック

以下の8項目を yes/no で判定する。no が出た項目は優先順位に従ってその場で修正する。

| # | キー | yes の条件 |
|---|------|-----------|
| 1 | **specificity** | 全ての色に hex 値、全てのフォントに family 名、全てのサイズに px 値がある |
| 2 | **roles** | 全ての色・コンポーネントに機能的役割（"Used for primary CTA" 等）がある |
| 3 | **completeness** | 9セクション全てに1項目以上の具体記述がある |
| 4 | **consistency** | セクション間で色名・フォント名・値が矛盾していない |
| 5 | **descriptiveness** | 技術値に自然言語の補足がある（"8px" だけでなく "やや丸み (8px)"）|
| 6 | **japanese** | 日本語コンテンツの場合、和文フォント・行間・禁則処理・OpenType が記述されている |
| 7 | **brevity** | トークンレベルに収まっている（デザインシステム全文になっていない）|
| 8 | **agent-ready** | Agent Prompt Guide に参照順序と迷ったときのフォールバック判断がある |

## 自己修正の優先順位

no が出た場合、以下の順で修正する:

1. **specificity** — 精密値の欠落は最も致命的。入力ソースに戻って値を取得する
2. **roles** — 機能的役割がないと使えない。「何に使うか」の一文を追加する
3. **completeness** — セクションが欠落している場合は補完する。情報が不足するなら「要補完」と明示
4. **consistency** — 矛盾箇所を入力ソースに照らして統一する
5. **descriptiveness** — 技術値に自然言語名を付与する
6. **japanese** — `references/japanese-typography.md` を参照して追記する
7. **brevity** — 冗長な記述をトークンレベルに要約する
8. **agent-ready** — 具体的な参照指示を Agent Prompt Guide に追加する

**迷ったら「曖昧にする」より「具体値を入れる」を優先する。**

## よくある失敗パターン

- `#1A73E8` だけ書いて名前と役割を省く → 使い所が分からない
- "Medium blue" のように名前だけで hex を省く → 実装できない
- 全コンポーネントを網羅しようとして冗長になる → トークンレベルに絞る
- VI PDF をそのまま転記する → 圧縮・変換が必要
- 情報が不足するセクションを推測で埋める → 「要補完」と明示する方が誠実
