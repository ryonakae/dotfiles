# Review Checklist

## 使い方

- スクリーンショットや動画があるなら、まず視覚的に確認する。
- 視覚材料がないなら、コード、画面構造、状態分岐、文言を見て判断し、未確認の視覚リスクを明示する。
- 指摘は重大度順に並べる。`タスク失敗 > OS 不整合 > 視覚的な粗さ` を基本順序にする。

## スクリーンショットで最初に見ること

- この画面の主目的が 3 秒でわかるか
- 主アクションが明確か
- 戻る、閉じる、保存、破棄などの基本操作が迷わないか
- 階層とグルーピングが自然か

## 操作性

- タップ対象が小さすぎないか
- 片手操作で厳しすぎる位置に重要操作がないか
- スクロール方向や画面遷移の期待が自然か
- キーボード表示時に入力中の要素や送信導線が隠れないか

## 可読性

- コントラストが足りるか
- テキストスタイルが多すぎないか
- 補助情報が本文より目立っていないか
- 余白不足や詰め込みで理解が遅くなっていないか

## 状態設計

- 読込中、空状態、成功、失敗、無効状態が設計されているか
- エラー時に次の行動が示されているか
- 破壊的操作に見合う確認や取り消し手段があるか

## OS らしさ

- iOS / Android の期待と大きくずれていないか
- ずらしている場合、その理由がブランドまたは機能要件として説明できるか
- 共通 UI にしている箇所が、単なる開発都合になっていないか

## ブランド整合

- 色、タイポグラフィ、モーション、文言のトーンが一貫しているか
- ブランドの主張が強すぎて、主要操作や可読性を邪魔していないか
- プロダクトの信頼感と世界観が両立しているか

## コードレビュー時の追加観点

- 状態ごとに UI が切り替わる実装になっているか
- デザイン都合の数値が散らばりすぎていないか
- OS 差分を吸収すべき箇所と共通化すべき箇所が整理されているか
- 文言、色、スペーシング、タイポグラフィが再利用可能な単位に寄っているか

## 典型的な指摘

- 主要操作が視覚的に弱い
- 空状態で次に何をすべきかわからない
- フォームの補助説明が不足している
- ブランド表現が強すぎて読みにくい
- Web 由来の構成でモバイルの操作文脈に合っていない
