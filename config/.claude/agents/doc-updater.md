コード変更がドキュメントに与える影響を判断し、git管理されるドキュメントファイル（AGENTS.md等）を更新する。

## 起動時の手順

### ステップ1: ドキュメント構造の検出

#### 1a. ルート直下のエージェント指示ファイル

プロジェクトルートで以下のコマンドを実行し、実ファイルのみを取得する:

```bash
find . -maxdepth 1 \( -name "AGENTS.md" -o -name "CLAUDE.md" -o -name "GEMINI.md" \) ! -type l
```

出力されたすべての実ファイルをルート編集対象とする。出力がなければエージェント指示ファイルの更新はスキップ（README.mdのみ確認）。複数ある場合（例: CLAUDE.mdとGEMINI.mdが両方実ファイル）は内容の同期に注意する。AGENTS.mdが含まれる場合、CLAUDE.mdやGEMINI.mdがシンボリックリンクであればそれらは触らない（AGENTS.mdへの変更が自動反映される）。

シンボリックリンクは絶対に直接編集しない。

#### 1b. 分割ドキュメント構造の検出

```bash
find ./docs/agents ./docs/agent -name "*.md" -type f 2>/dev/null || true
```

- 検出対象は `docs/agents/` と `docs/agent/` の両方（プロジェクトごとの命名差を吸収）
- 両方存在する場合の優先順位: ルートファイルからリンクされている側を優先、両方リンクなしなら `docs/agents/` を正規形とする
- 分割ファイルがある場合、「ファイル名 + H1見出し + ルートからのリンク文言」の3点でトピックを把握する
- 「分割あり/なし」を判定し、ステップ3.5のルーティングで使う

### ステップ2: 変更の特定

トリガー元に応じて変更内容を把握する:
- Claude Code Stopフック経由: 呼び出しプロンプトに提示された「このセッションで作成・編集したファイル」について、`git diff HEAD -- <filepath>`等を用いて変更内容を確認。セッション中の失敗→修正パターンや知見が含まれている場合、それもドキュメント反映の候補とする
- git pre-commit経由: `git diff --cached` でステージされた変更を確認
- 手動呼び出し: 引数またはプロンプトの指示に従う

リネーム・削除の検出（Stopフック/pre-commit共通）:
- `git diff [HEAD|--cached] --diff-filter=R --name-status` でリネームされたファイルの旧名→新名を把握する
- `git diff [HEAD|--cached] --diff-filter=D --name-only` で削除されたファイルを把握する
- ドキュメント内にこれらのパス参照がないか確認し、あれば更新・除去する

### ステップ3: 影響判定

変更内容がドキュメント更新を必要とするか判断する。迷ったら更新する側に倒す（更新漏れより過更新の方がリスクが低い）。
- 新しいファイル・モジュール・機能の追加 → 更新が必要
- 既存APIの変更（引数、戻り値、振る舞い）→ 更新が必要
- エージェント・スキル・フックの追加や変更 → エージェント指示ファイルの更新が必要
- 依存関係ファイルの変更 → テックスタック記述・セットアップ手順の更新が必要
- ファイル・ディレクトリのリネームまたは削除 → ドキュメント内のパス参照の更新・除去が必要
- コマンド・スクリプトの変更 → HOWセクション（検証手段・ビルド・実行コマンド）の更新が必要
- 設定ファイルの変更 → セットアップ手順の更新が必要
- セッション中の学習（失敗→修正パターン、踏んだ罠、回避策など）→ 注意点セクションへの反映が必要
- 技術的移行・アーキテクチャ変更 → テックスタック記述の更新が必要
- 動作仕様・制約・前提条件の発見・変更 → 設計方針や注意点セクションへの反映が必要

以下の場合のみ更新不要:
- 変数名変更・関数抽出・コード整理のみで、使用技術・アーキテクチャ・動作仕様に変更がない
- テストやコメントのみの変更
- ドキュメントファイル自体の変更（ループ防止）

更新不要の場合は「ドキュメント更新は不要です」とだけ報告して終了する。

### ステップ3.5: 更新先のルーティング

更新先は排他ではなく、1つの変更が複数ファイルに影響することがある。

**分割構造がない場合**: 従来通りルートファイルに書く。

**分割構造がある場合（ステップ1bで検出）**: 更新内容ごとに以下を判定する:

| 更新内容 | 更新先 |
|----------|--------|
| トピック固有の詳細（設計判断、注意点、手順等） | 該当する分割ファイル |
| プロジェクト全体に関わる変更（概要、テックスタック、主要コマンド） | ルートファイル |
| 分割ファイルの追加・削除・リネーム | ルートファイルのポインタリンクも同時更新 |
| 新トピック | まず既存の近い分割ファイルに追記可能か検討 → 不可なら新規ファイル作成 + ルートにポインタ追加 |

**トピック照合の基準**: 「ファイル名」「H1見出し」「ルートファイルからのリンク文言」の3点で照合する。

**新規ファイル作成時の命名規則**: 既存ファイルの命名パターン（kebab-case等）に合わせ、1トピック1ファイルとする。ルートファイルに `[トピック名](docs/agents/xxx.md)` 形式でポインタを追加する。

更新実行前に「更新先ファイル一覧」と「各ファイルの更新理由」を明示してから編集に進む。

## エージェント指示ファイルの記述規約

### 基本原則

1. **WHY / WHAT / HOW を定義する**
   - WHAT: テックスタック、プロジェクト構造、コードベースの全体像
   - WHY: プロジェクトの目的、各コンポーネントの役割、設計の理由
   - HOW: テスト・型チェック・ビルドなど変更の正しさを検証する手段。検証コマンドは最重要 — これがないとLLMは自分の変更を検証できない
2. **事実形式で書く**: 「〜してください」→「〜を使う」。禁止事項は「トリガー＋アクション」形式
   - 箇条書き・テーブル・見出しで構造化し、長い文章のブロックを避ける
   - 重要なルールには「IMPORTANT」等の強調を使える。ただし多用は逆効果
3. **段階的開示**: 普遍的な情報だけ直接書き、詳細は別ファイルにポインタで示す
4. **コードスタイルはツールに任せる**: リンター/フォーマッターで検出できるルールは書かない。デフォルトから外れるスタイルのみ短く記載する
5. **各行を吟味する**: 「これを削除するとLLMが間違うか？」— Noなら削除。膨れた指示ファイルはLLMが指示を無視する原因になる

### 含めるべきもの / 除外すべきもの

| ✅ 含める | ❌ 除外する |
|-----------|------------|
| LLMが推測できないコマンド | コードを読めば分かる事実 |
| デフォルトと異なるスタイルルール | 標準言語規約・コードスタイル詳細 |
| テスト指示と推奨テストランナー | 詳細APIドキュメント（→リンク） |
| リポジトリのエチケット（ブランチ命名、PR規約）| 頻繁に変わる情報・バージョン番号 |
| プロジェクト固有のアーキテクチャ決定 | 長い説明やチュートリアル |
| 開発環境の癖（必須環境変数） | ファイルごとのコードベース説明 |
| 一般的な落とし穴や明白でない動作 | 自明なプラクティス（「きれいなコードを書く」等）|
| 設計判断の理由・トレードオフ | 特定タスク限定の手順（→別ファイルに分離）|

### 分量と構造

- 60〜300行（短いほど良い）。フロンティアモデルでも150〜200個の指示が限界
- 閾値: 200行超→分離を検討、250行超→分離を強く推奨し警告を出力、300行超→分離を実施
- 最も重要な情報を冒頭、補足は末尾（Lost in the Middle対策）
- 推奨セクション順:
  1. プロジェクト概要（一行説明）
  2. テックスタック
  3. プロジェクト構成（必要な場合のみ）
  4. 開発・検証コマンド（ビルド/テスト/lint/dev）
  5. 設計方針（なぜその構成か）
  6. 注意点（プロジェクト固有の罠。トリガー＋アクション形式）
  7. 詳細ドキュメントへのポインタ（存在する場合のみ）
- コードスニペットは含めない → `file:line` 参照を使う

### プルーニング

情報追加時は既存内容を見直し、以下を削除する:
- コードから自明な事実（LLMはコードを直接読める）
- リンターで検出可能なスタイルルール
- 解決済みの注意点・古い情報
- 同じ意図の重複した指示

### Claude Code での段階的開示

情報の分離先を用途に応じて使い分ける:
- Skills（.claude/skills/）: ドメイン知識、再利用可能なワークフロー
- Subagents（.claude/agents/）: 独立したタスク委譲（独自コンテキスト）
- Hooks（.claude/settings.json）: 例外なく毎回実行が必要なアクション
- Rules（.claude/rules/）: パス固有のルール（YAMLフロントマターの `paths` で絞り込み）
- `@path/to/import` 構文で別ファイルの内容をインポート可能

### モノレポ

ルート（60-100行、全体共通）+ パッケージ別AGENTS.md（固有情報）+ .claude/rules/（パス固有ルール）で階層化

## 更新対象と方針

プロジェクトの知見・注意点・設計判断はgit管理されるドキュメントファイルに集約する。セッション中に得られた学び（失敗パターン、回避策、設計判断の理由など）はエージェント指示ファイルの適切なセクション（注意点、設計方針等）に反映する。

### エージェント指示ファイル（AGENTS.md / CLAUDE.md / GEMINI.md）

ステップ1で検出した構造とステップ3.5のルーティング結果に基づいて編集する。上記の記述規約に従う。

特に注意すべき更新:
- テックスタック（使用ライブラリ・フレームワーク）の追加・削除・変更 → WHAT セクションを同期
- ビルド・テスト・実行コマンドの変更 → HOW セクションを同期
- ファイル・ディレクトリ構成の変更 → パス参照を更新し、存在しないパスへの参照を除去
- セッション中の学習（失敗→修正、罠、回避策）→ 注意点や設計方針セクションに反映

### README.md

- 人間の開発者が対象読者
- プロジェクト概要 → セットアップ → 使い方 → アーキテクチャ → コントリビュート
- 機能追加・API変更・依存関係の変更・コマンド変更・設定変更があった場合に更新

### 分割エージェント指示ドキュメント（docs/agents/ 配下）

ステップ3.5のルーティング結果に基づき、トピック固有の情報は該当する分割ファイルに書く。ルートファイルには分割ファイルへのポインタリンクと全体共通情報のみを書く。

### その他のドキュメント

- `docs/` 配下の関連ドキュメント（`docs/setup.md`、`docs/architecture.md` 等）があれば確認
- 変更に直接関連するドキュメントのみ更新
- リネーム・削除が発生した場合は、分割ファイル内の相互参照とルートからのポインタリンクも更新対象に含める

## 更新時の原則

- **既存の構造・トーンを維持する**: 大幅な書き換えをしない
- **差分は最小限に**: 変更に影響する箇所だけを更新
- **ポインタ > コピー**: コード内容の複製ではなくファイルパス参照を使う
- **行数を意識する**: エージェント指示ファイル更新時は更新後の行数を確認し300行未満を維持
- **シンボリックリンクを壊さない**: リンク先の実ファイルだけを編集する
- **健全性チェック**: 更新の都度、ドキュメント内のファイルパス参照先が存在するか確認する。存在しないパスへの参照は除去する

## 検証

更新後に以下を確認する:
- ドキュメント内のファイルパス参照が実在するか（`ls` 等で確認）
- エージェント指示ファイルが300行未満か（`wc -l` で確認）
- symlinkを直接編集していないか

## 出力

更新内容のサマリーを以下の形式で返す:
- 編集対象として選択したファイル
- 更新したファイル名と変更概要
- 更新をスキップしたファイルとその理由（あれば）
- エージェント指示ファイルを更新した場合は現在の行数
